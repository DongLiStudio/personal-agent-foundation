# Obsidian 结构确认与连接

本流程只在用户配置真实 Obsidian Vault 时执行。目标是让 `GLOBAL/OBSIDIAN_LINK.md` 描述用户自己的知识结构，而不是复制作者或其他用户的目录约定。

## 一、先对话，不先扫描

先向用户确认：

1. 这是已有 Vault、空 Vault，还是准备新建的 Vault。
2. 用户是否已经有稳定的目录体系；如有，各主要目录分别承载什么。
3. 总览、索引或导航文件位于哪里。
4. 长期资料、项目资料、日记或临时记录、附件与媒体分别位于哪里；不存在的类别直接记为“不适用”。
5. Agent 默认可读取什么、必须排除什么，以及是否存在经用户明确授权的写入位置。
6. 是否有命名、标签、链接、归档或索引约定需要 Agent 遵守。

如果用户的回答已经足够，不要为了验证而额外扫描。

## 二、必要时做有界发现

只有用户同意后才能查看现有 Vault。发现范围默认限于：

- 列出 Vault 根目录一层的文件和目录。
- 查看用户点名的索引文件或目录一层。
- 确认用户明确指定路径是否存在。

不得递归遍历整个 Vault，不得进入 `.obsidian`，不得批量读取正文，不得跟随 Vault 内链接到其他位置。附件、私密日记、财务、身份资料等内容只有在用户明确点名并授权后才能读取。

文件夹名称不能单独证明其用途。发现结果与用户描述不一致时，应展示差异并继续询问，不要自行选择。

## 三、生成用户专属连接约定

基于已确认信息，拟定已安装实例中的 `GLOBAL/OBSIDIAN_LINK.md`。至少包含：

- Vault 实际绝对路径与 `GLOBAL/obsidian-resource` 入口。
- 已确认的用途到目录或文件映射；不适用项应明确写出。
- 建议优先阅读的总览或索引入口。
- 默认读写权限、允许写入位置和需要逐次确认的操作。
- 明确排除的目录或内容类型。
- 仍待用户决定的问题。

先把完整草案展示给用户。只有用户确认后，才替换已安装实例中的通用内容。不要修改产品仓库内的模板源文件。

## 四、创建与验收链接

目录映射确认后才创建链接：

- Windows 优先创建 Junction。
- macOS/Linux 创建 symlink。

创建前确认 Vault 真实存在，且不是 Agent 根目录、GLOBAL、它们的父目录或内部目录。创建后独立回读：

- 链接类型。
- `GLOBAL/obsidian-resource` 的实际路径。
- 解析后的 Vault 目标。
- `GLOBAL/OBSIDIAN_LINK.md` 的最终内容。

不得把 Vault 复制进 GLOBAL，也不得把创建链接等同于获得写权限。

## 五、特殊情况

- 未配置 Vault：不创建链接，保留通用说明并明确标记未配置。
- 空 Vault 或准备新建：询问用户希望保持空库还是建立建议结构；展示方案并取得单独确认后才能创建目录。
- 用户采用作者默认结构：仍展示识别出的映射并确认，不能仅凭同名目录自动接受。
- 用户暂时说不清结构：完成其他安装步骤，把 Obsidian 标记为待配置；不要用猜测内容完成验收。
