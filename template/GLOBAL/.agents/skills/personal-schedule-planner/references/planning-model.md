# 排程决策模型

## 观察与排程范围

- 默认实际排程：当前时刻至今天结束。
- 默认观察范围：未来七天，用于发现截止、依赖和冲突；不要因此自动创建未来七天的全部时间块。
- 用户指定未来几天、本周或其他日期范围时，以用户范围为准。

## 候选工作项

每个候选项至少保留：

- `source_ids`：任务 GUID、项目任务文件或 Obsidian 笔记定位；不把秘密写入日历。
- `source_profiles`：在哪些飞书 Profile 中出现。
- `project`：所属 Agent 项目；无法判断时标为未归类，不强行归属。
- `title`：草案与全部目标日历共同使用的唯一标题。
- `start`、`due`、固定日程、依赖和阻塞。
- `duration`：用户给定或有依据的估计；同时记录置信度。
- `importance`、`urgency`、`strategic_value`、`energy` 与可拆分性。

## 归一化与去重

1. 相同飞书任务 GUID 视为同一任务，合并 Profile 来源。
2. 项目 `STATUS.md`、Obsidian 和飞书任务指向同一明确交付时合并为一个候选项，并保留全部证据来源。
3. 标题相似但交付、负责人或截止不同的任务不得合并。
4. 同一固定日程跨 Profile 重复出现时，优先依据本 Skill 稳定标记识别；没有标记时只合并有充分证据的同一事件。
5. 无法可靠判断是否重复时保留候选并在草案中标注，不擅自删除或合并。

## 约束层级

按以下顺序处理：

1. 已存在的会议、预约、确定性通勤、用餐及其他不可移动承诺。
2. 已逾期或即将到期、会阻塞他人或后续工作的交付。
3. 用户明确强调的当天目标和高战略价值项目推进。
4. 适合当前精力和连续时间的深度工作。
5. 可批处理的沟通、审批、整理和低认知杂项。
6. 可延期、低影响或证据不足的候选项。

不要用机械总分替代判断。出现截止更近但价值较低、战略价值高但无期限等冲突时，在草案中说明取舍。

## 时间块规则

- 固定日程先占位，再在所有 Profile 忙碌时间并集之外排工作。
- 深度工作通常使用连续块；沟通和杂项合并处理，减少上下文切换。
- 预计时长低置信时留出额外缓冲，或把任务拆为“调查/确认”和“实施”两个可验收块。
- 默认只安排约 70%–80% 的真实可用时间，其余用于休息、切换、临时沟通和延误；紧急情况下可以提高，但必须明确风险。
- 吃饭可作为受保护时间块进入日历；普通休息、切换缓冲、延误余量和突发容量使用 `action: none` 留白，不创建日历事件。
- 确定性通勤可作为受保护时间块进入日历；仅用于容错的通勤弹性仍使用 `none`。
- 时间块不得重叠，开始必须早于结束，且不得落入已经完全过去的时间。
- 不把“尽快”“有空”“今天看看”等模糊语言自行转换为硬截止。

## 标题与内容边界

- 草案只维护一个 `title`，同步到全部目标 Profile 时原样使用。
- 不在标题或描述中放入客户秘密、金额、人员隐私、未发布敏感信息、内部链接或任务全文。
- 若原始事项名称含不宜公开的细节，在进入草案前缩减这些细节，并由用户确认最终唯一标题。

## 草案格式

草案至少包含：

| 时间 | 安排 | 依据与取舍 | 动作 |
|---|---|---|---|

表格后列出：

- 未排入任务及原因。
- 截止风险、依赖和冲突。
- 时长或优先级推定。
- 保留缓冲。
- 目标 Profile 与逐块 `create/update/retain/delete/none` change set；`none` 明确表示留白而非待创建事件。

每次调整后重新输出完整版本，不让用户通过多轮增量自行拼接最终计划。
