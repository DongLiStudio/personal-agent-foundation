# 全局与项目局部路由

## 三层事实模型

1. 全局 Skill：维护通用判断、CLI 生命周期流程和安全规则。
2. 项目文件：维护该项目使用哪个主体、何时切换、哪些资源属于例外。
3. CLI/系统密钥链：维护 App Secret、tokens 和实时登录状态。

三层不得复制彼此的职责。

## 放置判断

| 信息 | 放置位置 |
|---|---|
| 新增、授权、恢复、换机、重命名、删除的方法 | 全局 `feishu-profile` Skill |
| 用户决定全局维护的公司/Profile 非秘密映射与全局默认 Profile | `GLOBAL/LARK_PROFILES.md` |
| 项目需要覆盖 GLOBAL 默认时的默认 Profile 与强制 user/bot 边界 | 项目 `AGENTS.md` |
| 仅服务某项目的主体、客户租户或专项 Profile 例外 | 项目资源路由文档 |
| Base/Wiki/Table 等项目专属非秘密定位 | 项目资源路由文档或业务文档 |
| App Secret、token、device code、认证缓存 | 仅 CLI/系统安全存储；不进项目和 GLOBAL |
| Profile 是否当前有效、当前登录用户 | 不持久化；每次实时回读 |

归属首先服从用户明确决定。用户本人或用户控制、需要长期使用的飞书组织通常登记到 `GLOBAL/LARK_PROFILES.md`；客户、合作方、第三方、测试、沙箱或专项身份通常保留在相关项目。是否跨多个项目使用只是辅助依据，不构成硬门槛。进入 GLOBAL 只表示统一维护非敏感映射，不自动设为全局默认，也不改变 CLI active Profile。

## 项目标准结构

### `AGENTS.md`

项目默认 Profile 是可选配置。项目没有覆盖规则时，继承 `GLOBAL/LARK_PROFILES.md` 记录的全局默认 Profile，并在操作时显式使用；CLI active 只用于核验。项目需要覆盖或增加例外时，保持简短：

```md
## 飞书项目身份

- 本项目默认 Profile 为 `<名称>`，用于覆盖 GLOBAL 默认；默认身份为 `user`。
- 需要使用项目默认时，所有命令显式使用 `--profile <名称> --as user`。
- 用户明确要求操作其他主体时，按对应资源路由规则执行；不得静默回退全局默认或 bot。
```

### 条件化客户路由

推荐位置：`docs/clients/<客户>/飞书资源路由.md`。记录：

- 服务提供方与客户主体关系；
- 客户 Profile 的逻辑名称；
- 何种明确条件下才能使用；
- 允许操作的资源类别与非秘密定位键；
- 禁止写入的本项目内部资源；
- 写前写后的 `whoami`、资源解析与回读要求；
- 当前是否已完成配置，只记录结论，不记录 token 或历史个人 ID。

## 维护动作

- 新增项目局部身份：先确认主体、用途和资源归属，再更新路由文档；不改变项目默认 Profile。
- Profile 重命名：只更新当前授权项目中的精确引用；其他项目列为待处理，不跨项目静默修改。
- 删除局部身份：先移除或改写项目引用，再删除 CLI Profile；保留必要历史事实。
- 项目局部身份升级为全局映射：以用户明确决定为前提，结合组织归属、长期用途和项目引用登记非秘密映射；不以项目数量作为硬门槛，项目仍保留自己的使用条件。
- 全局映射降级为局部：列出其他项目引用和影响，取得用户明确确认后才能从 GLOBAL 移除；不得自动降级。凭据状态不随文档删除自动变化。

## 不创建局部 Profile Skill

Profile 路由本身是项目规则，不创建 `.agents/skills/<project>-profile`。只有独立业务流程具备自己的触发条件、数据模型和交付步骤时，才创建项目业务 Skill；其认证异常统一转交全局 `feishu-profile`。
