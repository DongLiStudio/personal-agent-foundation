# 人员角色判定与场景矩阵

## 角色判定顺序

1. 标注每句话的说话人和消息时间。
2. 分别识别任务提出者、交办者、实际执行者、协作者、审核者、验收者和仅需知悉者。
3. 按“用户当前明确指定 → 最终确认分工 → 行动词归属 → 默认规则”覆盖合并。
4. 先确定实际工作动作，再决定 assignee；不要先按聊天参与人分配角色。
5. 最后补充 follower，并保证当前登录用户始终在关注人中。

## 负责人规则

- “负责、制作、整理、提交、跟进、协调、协助、审核”等行动只有构成本任务必要交付时，相关人员才作为负责人。
- “B 负责”应移除不再执行的默认负责人；不要因为用户创建任务而继续把用户保留为负责人。
- “我和 B 一起做”“B 做素材，我负责生成”应加入所有实际承担交付的人，并写清分工。
- “协助、配合、提供素材、审核、验收”若只是外围关系，不加入负责人；若是必需任务动作，则加入并说明动作。
- 同事仅被提及、发言、作为客户、信息来源或会议参与人，不加入负责人。

## 关注人规则

- 当前登录用户始终为关注人。
- 提出、交办、要求、委托或转述任务要求的人默认加入关注人。
- 明确审核、验收、知悉、同步、抄送或跟进可见的人加入关注人。
- 任务提出者默认是关注人，不因提出任务自动成为负责人。
- 仅参与无关讨论或被顺带提及的人不加入关注人。

## 常见场景矩阵

| 上下文 | 负责人 | 关注人 |
|---|---|---|
| 用户自己记录任务，未提及执行者 | 当前登录用户 | 当前登录用户 |
| 同事 A 让用户做 | 当前登录用户 | 当前登录用户、A |
| 用户让同事 B 做 | B | 当前登录用户 |
| 同事 A 对用户说“让 B 做” | B | 当前登录用户、A |
| 同事 A 让用户和 B 一起做 | 当前登录用户、B | 当前登录用户、A |
| 同事 A 让 B 和 C 一起做 | B、C | 当前登录用户、A |
| 同事 A 让 B 做、C 协助必要交付 | B、C | 当前登录用户、A |
| 同事 A 让 B 做，并要求同步给 C | B | 当前登录用户、A、C |
| 同事 A 让用户协调、B 执行 | 当前登录用户、B | 当前登录用户、A |
| 同事 A 仅建议“可以考虑让 B 做” | 不据此确定 B；使用其他明确分工或询问 | 当前登录用户、A |
| 用户更正“不是我做，让 B 做” | B | 当前登录用户；按来源补充提出者 |
| B 只提供参考资料，不承担交付 | 其他明确执行者；无明确执行者时为当前登录用户 | 当前登录用户；仅在需同步时加入 B |

矩阵是默认解释，用户明确指定的最终分工优先。同一人可以同时是负责人和关注人。

## 必须询问的情形

- 截图左右气泡、头像或昵称不足以确定说话人。
- 代词“他、她、你们、那边”存在多个合理指向。
- 同名联系人或姓名解析得到多个候选。
- 既可理解为执行者，也可理解为审核/知悉者，且会改变任务责任。
- 前后分工冲突，无法判断哪条是最终决定。

只询问决定写入所必需的信息，不要求用户重新复述全部任务。
