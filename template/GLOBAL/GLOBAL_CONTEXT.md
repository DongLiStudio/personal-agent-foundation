# Agent 全局上下文

本文件是任何 Agent 会话的稳定全局入口。

- 固定工作区：`{{AGENT_ROOT}}\GLOBAL`
- 全局用途：保存跨项目、跨会话、跨 Agent 工具的规则、项目路由和长期协作约定。

## 工作区模型

Agent 工作采用项目制管理。

- `GLOBAL` 是治理层，应保持小、稳、可复用。
- 项目目录是工作层，项目专属任务、草稿、文档、代码、资产和决策进入对应项目。
- 不要把活跃项目工作直接堆进 `GLOBAL`。

当一个长期目标出现时，优先创建或选择项目目录，再开始积累文件。

## 标准项目结构

项目初始化时默认创建：

```text
项目名/
  README.md
  STATUS.md
  AGENTS.md
  .gitignore
  docs/
    .gitkeep
  tasks/
    .gitkeep
  archive/
    .gitkeep
```

`STATUS.md` 记录项目现状，`README.md` 记录项目身份和入口，`AGENTS.md` 记录 Agent 工作规则。

`.gitkeep` 只用于让 Git 跟踪空语义目录；当目录中填充了实际内容，应移除对应 `.gitkeep`。

## Skill 源头模型

`{{AGENT_ROOT}}\GLOBAL\.agents\skills` 是个人全局 Skill 的源稿目录，不是 Agent 运行时安装目录，也不是项目内部 Skill 默认目录；用于维护、审查、同步和迁移。

- 用户维护、跨项目复用的 skill 源稿放在这里。
- 项目内部 Skill/agent 内容使用 `.agents/` 机制。
- 不要把系统 skill、插件 skill、缓存或运行时安装产物复制进 `GLOBAL`。
- 全局 Skill 恢复遵循 `SKILL_DEPENDENCIES.md`。

Obsidian 连接、目录理解和优先阅读入口见 `OBSIDIAN_LINK.md`。

## 项目组织

- 每个项目默认设立“项目总经理”职责，负责整体判断、任务拆分、长期岗位任命、结果验收和跨项目汇报；同一人或任务窗口可以兼任多个职责。
- 用户明确任命或 GLOBAL 明确授权按需补充的长期岗位、稳定职责及协作边界，统一记录在项目 `AGENTS.md` 的“项目组织”模块中；用户任命优先，不根据窗口名称或当前任务自行推断组织关系。
- 岗位补充不是只改 `AGENTS.md`。当规则要求新增或复用长期岗位时，完成条件必须同时包含：项目 `AGENTS.md` 已登记岗位职责、已创建或复用对应长期会话/任务窗口、已把岗位启动指令和当前上下文交给该窗口、该窗口已回报接单或阻塞。环境暂不支持创建/定位长期窗口时，必须明确报告“岗位规则已登记，执行窗口待创建/待用户接管”，并把它作为待办或阻塞边界，不得把单纯写入 `AGENTS.md` 宣称为岗位已落地。
- 长期任务窗口承接持续职责；当前任务中临时创建的子智能体只承担本次委派，不自动成为项目长期岗位。
- 每个长期岗位默认有权按任务需要自行创建和使用会话内临时子智能体，无需逐次申请；临时子智能体只能承接该岗位本来可以执行的工作，不扩大任务范围或操作权限。长期岗位仍负责委派判断、结果验收和最终回报。
- 长期岗位完成每次派发任务后，必须主动向项目总经理回报结果、验证、提交或外部操作、遗留问题和建议下一步；不得只在自己的任务窗口中结束。
- 每个项目首次需要持续创建或维护飞书任务时，若“项目组织”中没有承担同等职责的岗位，按需补充“项目任务协调员”：只管理本项目任务，遵循本项目 Profile、负责人和授权规则，并把影响当前用户排程的变更通过标准 handoff 交给{{GENERAL_ASSISTANT_PROJECT}}。已有等价岗位时复用，不重复设置。
- {{GENERAL_ASSISTANT_PROJECT}}首次需要跨项目个人排程时，若“项目组织”中没有承担同等职责的岗位，按需补充“个人时间管理员”：汇总各项目 handoff、全部治理层飞书 Profile、Obsidian 仪表盘和项目状态，负责排程协商及确认后的日历同步，但不改变项目任务所属 Profile、负责人或业务内容。已有等价岗位时复用，不重复设置。
- 上述两个岗位属于用户已确认的按需组织规则，不需要为每次补充重复询问。环境支持长期任务窗口管理时，必须建立或复用对应窗口并指派；暂不支持时由当前会话完成本次工作并报告岗位登记状态与“长期窗口未创建”的待办，不得因缺少可调用窗口而阻塞本次任务，也不得隐瞒岗位尚未具备持续承接窗口。
- “项目组织”模块默认包含项目总经理、用户已明确任命或按上述规则补充的其他长期岗位及通用协作规则；不得为了结构完整虚构其他岗位，也不记录任务窗口 ID、临时忙闲、单次派工或组织变更流水账。

## 协作规则

- `GLOBAL` 提供上层规则；具体任务优先读取具体项目的 `README.md`、`STATUS.md`、`AGENTS.md` 和相关任务文件。
- 各项目原则上独立维护自己的 Git 仓库和版本历史；完成可验证的阶段性成果后，按项目规则创建聚焦、可回退的提交，提交前检查 diff、相关验证和敏感信息，不跨项目暂存或混入无关改动。commit 不等于 push，推送和发布仍按外部操作规则处理。
- 如果已配置全局默认 GitHub 账号，且项目没有指定专属账号，项目需要远程备份时优先使用全局默认账号；项目存在专属账号或路由规则时，以项目规则为准。首次创建远程仓库、确定仓库名与可见性、配置 remote 和首次 push 前必须获得用户明确授权，不能因存在本地 commit 或默认账号而自动上传。
- 只有任务确实依赖长期个人知识、战略、资源、业务背景或既有规划时，才搜索或读取 Obsidian。
- 搜索本地上下文时优先使用 `rg`。
- 新笔记或重大修改先在对应 Agent 项目工作区起草。
- 写入 Obsidian 前必须询问。
- 输出应务实、项目化，并连接用户已有系统。
